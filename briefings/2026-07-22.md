## Changelog digest

`CHANGELOG.md`'s top section is now dated **2026-07-22** and carries eight entries, all touching `01-lya_halos` directly. In roughly the order the work happened (the CHANGELOG lists newest-first, so this reverses it):

- **The root cause — Lyα PSF forward model had the wrong flux convention.** `fitting.bin_average_psf`/`bin_average_psf_expcore` computed `(R @ [I(r)*dr]) / bin_width` — input ring flux missing its `2*pi*r` circumference, output divided by radial width instead of annulus area. Nearly harmless at a narrow PSF (the two errors cancel within a bin), catastrophic once a realistically wide PSF makes the ring-convolution matrix mix flux across radii — which is exactly the "wide-PSF h1 collapse" that had been an open puzzle. **This is the identical defect fixed in the UV path on 07-15/17 (`bin_average_psf_uv_exp`), simply never back-ported to the Lyα models.** All four Lyα entry points now use `(R @ [I(r)*2*pi*r*dr]) / annulus_area`; the two UV *naive* functions were fixed at the same time (they were still on the old width convention while their PSF counterparts weren't, so `method='naive'` and `method='psf'` UV fits weren't measuring the same quantity). `ring_convolution_matrix` itself is unchanged and still passes `validate_ring_convolution`.
- **Diagnosis + verification harness — `01-lya_halos/code/psf_forensics.py`** (new, standalone, does not import into the pipeline). Four tests against real data plus a brute-force 2-D image-convolution reference: the corrected form agrees with brute force to <1% at both 1.3″ and 2.4″ PSF FWHM while the old form was off by **51% and 96%**; the deconvolvability check **passes for both PSF widths, so the 2.4″ measurement was never the problem**; the fit ladder gives a stable `h` (15.14±1.03 kpc at 1.3″, 12.02±0.99 kpc at 2.4″) under the corrected convention while the old one collapses to h→0.011 at 2.4″; and an FWHM sweep 1.0″–3.0″ shows smooth `h = 15.9→10.1 kpc` with **no cliff anywhere**.
- **Radial quadrature rule fixed — 113× more accurate at the same cost.** The ~2.8% innermost-bin error that looked like "not enough grid points" (`n_per_bin`) was actually a quadrature bug: `default_fine_grid` jumps grid spacing at every bin edge, and `np.gradient` (central difference) + `np.add.at` silently dropped each bin's outer endpoint where its `2*pi*r` weight is largest. Replaced with a per-bin composite trapezoid rule (`_segment_weights`, `_bin_integrate_2d` on `np.trapezoid`). **Innermost-bin error 2.772% → 0.0245% at the unchanged `n_per_bin=30`** — better than `n_per_bin=100` was under the old rule, at zero runtime cost.
- **New `measure.measure_psf_corrected_core_onehalo_luminosity` — fixes the core↔one-halo double-count.** `measure_psf_corrected_core_luminosity`'s own model says a fraction `(1-EE_i)` of each galaxy's reconstructed point-source flux scatters *outside* the core aperture — but `measure_onehalo_luminosity` sums that same radial range from the raw stack, so that light was counted twice (boosted into the core, and still sitting raw in the one-halo zone). The new function extends the per-galaxy Moffat encircled-energy model from a single `EE_i` to a full profile at every one-halo bin edge, and subtracts each bin's leaked fraction from that galaxy's raw core spectrum *before* stacking (valid because the PSF smears position, not wavelength). Core and one-halo now come out of one restack + one bootstrap pass. Returns `(core_lum, onehalo_lum)`, key-compatible with the two existing functions so it drops straight into `measure_three_zone_ratios`/`describe_subsample_properties`; carries `onehalo_lum_raw_*`/`leaked_lum_*` diagnostics for a before/after check. The two older functions are unchanged and remain the cheaper call when only one zone is needed.
- **Real bug found inside that fix — missing area-normalization in the default per-bin mode.** An injection-recovery test (built at Austin's request) caught it: the per-bin leak subtraction happens in surface-brightness space against the core aperture's area, but the result is converted to luminosity using each zone bin's *own* area, so the amount removed was off by `area_zone_j / core_area_kpc2` per bin (~70× too large for an outer bin). Symptom: a known-truth synthetic returned a **negative** one-halo luminosity (−137% off), while the fast mode (never had this step) recovered truth to <0.1%. One-line fix (`leak_ratio = (leak_frac / EE_core[:, None]) * (core_area_kpc2 / area_zone)[None, :]`); the test now recovers both core and one-halo to 0.0% in both modes. **Every prior run of the default (bin-by-bin) mode is affected and should be re-run;** the fast-mode numbers were never affected.
- **~4× speedup, no change to results.** `bootstrap_diagnostics` (new kwarg, default `False`) stops restacking the raw one-halo zone every draw just to populate a diagnostic — the fiducial version now comes free from `boot['total_flux_fid']`. The per-bin blue/red flux extraction was vectorized across all columns at once for `cont_method='median'`. Verified numerically identical (`rtol=1e-8`); benchmarked 25.15s → 6.52s per 100 draws at realistic scale (450 galaxies, 7 bins).
- **Measurement-window crop + `assume_no_leak_beyond_boundary` fast mode.** `cProfile` showed **88% of runtime inside `astropy.stats.biweight_location`** — the earlier vectorization was optimizing a ~5–10% slice. The cube is now sliced to `_measurement_window` before restacking (the same helper `bootstrap_all` already uses). New opt-in `assume_no_leak_beyond_boundary=True` reduces the one-halo correction to pure flux conservation (`onehalo_true = onehalo_raw − (core_lum − raw_core_lum)`), restacking only 2 columns per draw instead of `1+n_zone`. Also surfaced: the leaked-flux fraction swings **~2–5× across the 1.3″↔2.4″ PSF-width range** (e.g. 29%→64% at R_inner=10 kpc) — the correction's *size* carries a real, currently-unquantified PSF-width systematic.

**Consequences to act on (Austin already flagged these in the CHANGELOG, repeated here so they don't get lost):** (a) every PSF-aware Lyα fit and the default-mode one-halo luminosity run before today needs re-running — `h1`, crossover radii, zone luminosities, and all subsample splits; (b) fitted amplitudes `A1`/`A2` are now central *surface brightnesses* — absolute pre-fix and post-fix values aren't comparable (scale lengths, crossover radii, and amplitude *ratios* are); (c) the r≈48 kpc residual previously flagged as a separate anomaly appears to have been this same bug; (d) the extension is detected overwhelmingly (adding `h` drops χ² by ~173); (e) the data only mildly prefer 1.3″ over 2.4″ (Δχ²≈1 on 3 dof), but that choice moves `h` from ~15.1 to ~12.0 kpc — **PSF width is now a systematic to quote, not a bug to chase.**

## Literature scan

Checked against all current filenames in `docs/literature/` (52 files) to avoid repeats, and against the open suggestions carried in the 07-10/07-13/07-17/07-21 briefings (Sebastian 2026, Li 2025, Gagnon-Hartman 2026, Momose 2014 — now in the library — plus Smirnov 2026, Claeyssens 2022, Verhamme 2018 — none re-listed here). Weighted toward this week's actual theme: **PSF forward-modeling / encircled-energy corrections in stacked Lyα profiles**, and the core-vs-one-halo-vs-two-halo luminosity decomposition the double-count fix is protecting.

**Recent (2025–2026):**

1. **Cerdosino 2025** — ODIN: Using multiplicity of Lyman-alpha emitters to assess star formation activity in dark matter halos (arXiv:2511.01981, Nov 2025). Uses the ODIN narrowband survey to identify physically associated LAE multiples in COSMOS at z=2.4/3.1/4.5 and tests whether multiplicity traces host halo mass, backed by an IllustrisTNG100 mock for completeness/contamination. Finds halo-wide Lyα and UV surface-brightness densities rise with multiplicity, and that a subhalo-perturbation SF model (not companion presence per se) drives the activity — a direct, environment-side handle on the same two-halo/clustering term this project's `measure_twohalo_luminosity` and the overdensity subsample splits are built to quantify from the emission-stacking side.

2. **Yu 2025** — Lyα radiative transfer modeling for 163 MUSE Lyα-emitting galaxies at z=3–6 (arXiv:2506.08290, June 2025). Forward RT models both the Lyα spectrum *and* the surface-brightness profile *simultaneously* for 163 HUDF LAEs, and correlates observables against model parameters — finding the halo's spatial extent is set primarily by the extents of the medium and the source, while spectral peak-shift/FWHM are driven by separate physics. Relevant less for its numbers than its methodology: an independent full-RT route to the same "which physical component sets the profile scale" question the core/one-halo/two-halo split is answering empirically, and a natural external check on the PSF-corrected `h1`.

**Older, likely not yet reviewed:**

3. **Davis 2023** — HETDEX Public Source Catalog 1: Stacking 50,000 Lyman Alpha Emitters (arXiv:2307.03096, ApJ, July 2023). Stacks ~50k spectroscopically selected LAEs at 1.9<z<3.5 from HETDEX, using the ensemble to recover spectral and spatial features hidden in individual noise. The most direct large-N methodological analog to this project's own stacking pipeline — a useful comparison point for stack-level systematics and PSF handling, and a natural benchmark now that the Lyα PSF forward model has been corrected.

4. **Xue 2017** — The Diversity of Diffuse Lyα Nebulae around Star-Forming Galaxies at High Redshift (arXiv:1611.03510, ApJ, 2017). Detects diffuse Lyα halos around ~1400 galaxies in two spectroscopically confirmed overdensity regions in the Boötes field at z≈2.66 and 3.78, and explicitly checks that different stacking methods return consistent scale lengths. Directly relevant on two fronts: it's an environment/overdensity split (the Chartab-density plan's exact axis) *and* a stacking-method-robustness test of the kind worth citing when defending this pipeline's own PSF/stacking choices in the Methods section.
