# Specs

Design docs written *before* building a notebook, pipeline stage, or analysis. Each spec should cover:

- **Goal** — what question this is answering
- **Data** — which catalogs/images/products it uses
- **Method** — the approach, cuts, or algorithm
- **Success criteria** — how you'll know it worked

Keep one file per analysis/stage (e.g. `stacking-v2.md`, `virial-mass-estimator.md`). Update the spec if the approach changes significantly — it should stay a reasonably accurate record of "what we're doing and why," not just a historical note.
